GraphOS schema linter allows you to enforce consistency in schema design with automation that can check for formatting, styling and naming conventions that adhere to GraphQL and federation best practices.

## Configure Lint Checks in Studio

To configure lint checks in Studio, navigate to the Linter tab from the Checks page.
Under Default Linter Configuration you will see the default linter settings for variants of this graph. Enabling the linter will run the linter on all graph variants. Note that the linter will only run on modified parts of your schema and not the entire schema.

You can choose to ignore rule violations on @deprecated definitions and @inacessible definitions.

The next section of this page contains the list of linter rules with their name, description, and severity level.

The severity level defaults to Error but you can use the toggle to configure to a lower severity, warning.

**Rule name:**
`ENUM_PREFIX`

**Description:**
Enum type names should not be prefixed with the word Enum.

**Example:**

```graphql
# :x:
enum EnumLocationType { # highlight-line
    SPACESHIP
    HOUSE
    CAMPSITE
}

# :white_check_mark:
enum LocationType { # highlight-line
    SPACESHIP
    HOUSE
    CAMPSITE
}
```

**Rule name:**
`ENUM_SUFFIX`

**Description:**
Enum type names should not be suffixed with the word Enum.

**Example:**

```graphql
# :x:
enum LocationTypeEnum { # highlight-line
    SPACESHIP
    HOUSE
    CAMPSITE
}

# :white_check_mark:
enum LocationType { # highlight-line
    SPACESHIP
    HOUSE
    CAMPSITE
}
```

**Rule name:**
`ENUM_USED_AS_INPUT_WITHOUT_SUFFIX`

**Description:**
Enum is used as an input argument. Suffix Enum type name with the word Input.

**Example:**

```graphql
# :x:
type Query {
    directions(location: LocationType) # highlight-line
}


# :white_check_mark:
type Query {
    directions(location: LocationTypeInput) # highlight-line
}
```
